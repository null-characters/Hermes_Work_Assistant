## 一、架构契合度分析

### 1.1 现有架构 vs 规划架构对比

| 维度 | 现有 Hermes Work Assistant | 规划 Hermes-ERP |
|------|---------------------------|-----------------|
| **交互层** | Streamlit Web UI (8501) | 微信/企微 Gateway |
| **决策层** | Hermes Agent (Docker 容器) | Hermes Agent + MCP Tool Calling |
| **执行层** | Hermes Bridge (FastAPI 8646) | Kingdee MCP Server |
| **存储层** | 本地文件系统 + SQLite | 本地 + ERP 云端数据 |

### 1.2 架构演进路径评估

**现有架构优势**：
1. ✅ **会话隔离机制成熟**：`data/sessions/{session_id}/` 目录结构可直接复用
2. ✅ **安全验证完备**：`validate_path()` 白名单、`validate_prompt()` 黑名单已实现
3. ✅ **流式响应机制**：`HermesClient.execute_task_stream()` 支持实时思考过程展示
4. ✅ **Docker 容器化**：Agent 已容器化，便于扩展

**架构差距**：
1. ❌ **无 MCP 协议支持**：当前 Hermes Agent 通过 `hermes chat` CLI 交互，未暴露 MCP 接口
2. ❌ **无微信/企微网关**：当前仅 Web UI，缺少 IM 接入层
3. ❌ **ERP SDK 未集成**：`kingdee_webapi_sdk` 为独立项目，未封装为 MCP Server

## 三、技术风险评估

### 3.1 高风险项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **MCP 协议版本兼容性** | Hermes Agent 可能不支持最新 MCP 规范 | 使用 MCP Inspector 验证，必要时降级 SDK 版本 |
| **ERP API 认证泄露** | API 凭证被 LLM 窃取 | 凭证封死在 MCP Server `.env`，LLM 只传业务参数 |
| **大数据查询 Token 爆炸** | 10 万条数据撑爆 LLM Context | MCP Server 强制分页（limit=100），大数据写临时文件 |

### 3.2 中风险项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **微信网关稳定性** | 社区项目维护风险 | 优先使用 Hermes 官方 Gateway，备选 HermesClaw |
| **ERP 写操作误判** | 用户意图理解错误导致错误建单 | 强制 Grounding：写操作前必须查询确认 ID |

## 四、架构演进建议

### 4.1 推荐演进路径

```
Phase 1 (MVP)          Phase 2           Phase 3           Phase 4
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ Kingdee     │    │ Hermes      │    │ WeCom       │    │ Skills      │
│ MCP Server  │───▶│ Agent + MCP │───▶│ Gateway     │───▶│ Learning    │
│ (Python)    │    │ Integration │    │ Integration │    │ System      │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
     1-2周              1周               1-2周            中长期
```

### 4.2 关键架构决策

| 决策点 | 建议方案 | 理由 |
|--------|----------|------|
| **MCP SDK 选型** | `mcp-python-sdk` 官方库 | 社区活跃，文档完善 |
| **Hermes Agent 改造** | 注册 External MCP Server | 最小侵入，复用现有 Agent |
| **ERP 凭证管理** | MCP Server 环境变量 | 与现有 `.env` 机制一致 |
| **会话数据隔离** | 复用现有 SQLite | 无需引入新存储 |

## 五、结论

### 5.1 架构可行性评估

| 维度 | 评分 | 说明 |
|------|------|------|
| **技术可行性** | ⭐⭐⭐⭐ | MCP 协议成熟，SDK 已有 Python 实现 |
| **架构契合度** | ⭐⭐⭐⭐ | 现有会话隔离、安全机制可直接复用 |
| **风险可控性** | ⭐⭐⭐ | 主要风险在 MCP 兼容性，需验证 |
| **实施复杂度** | ⭐⭐⭐ | 中等复杂度，需新增 MCP Server 模块 |

### 5.2 核心建议

1. **优先验证 MCP 兼容性**：使用 MCP Inspector 测试 Hermes Agent 是否正确发现和调用工具
2. **最小化 Hermes 改造**：通过 External Server 注册而非修改 Agent 核心
3. **安全第一**：ERP 凭证绝不暴露给 LLM，所有写操作需二次确认
